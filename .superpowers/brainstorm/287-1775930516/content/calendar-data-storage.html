Decision: Single date field vs separate date + time fields?
Option A — Single date field (recommended)
One date field stores full ISO 8601 timestamp (e.g. 2026-04-15T09:30:00+10:00). "All day" = 2026-04-15T00:00:00. Simpler schema. Consistent with SQLite date storage.
Apple Calendar uses this internally. No separate time field needed.
Option B — Separate date + time fields
date = 2026-04-15 (YYYY-MM-DD string). time = 09:30 (HH:MM string). Explicit separation.
Google Calendar often separates these in their API.

Import / Export — ICS (.ics) support
📄 ICS (.ics)
Universal standard. Every calendar app supports it — Apple Calendar, Google Calendar, Outlook, Thunderbird, Notion, Linear.
Text format. Human-readable. Easy to debug. Version 2.0 is the current standard (RFC 5545).
BEGIN:VCALENDAR BEGIN:VEVENT DTSTART:20260415T093000 DTEND:20260415T110000 SUMMARY:Write API docs RRULE:FREQ=WEEKLY END:VEVENT END:VCALENDAR
🌐 CalDAV (WebDAV)
Two-way sync. Real sync protocol — not just import/export. Changes propagate both ways.
Complex. Requires a CalDAV server (Apple iCloud, FastMail, ownCloud, Baïkal). Harder to implement. Overkill for simple import/export.
Use case: If users want Handy to sync directly with Apple Calendar or Google Calendar in real-time.
🗓️ Google Calendar API
One-way or two-way. Google Calendar API for direct push/sync to Google Calendar. OAuth required.
Proprietary. Only works with Google. App-specific. Good for "Export to Google Calendar" button.
Use case: "Sync to Google Calendar" with one-click OAuth.
