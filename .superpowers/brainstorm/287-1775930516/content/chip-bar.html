CalDAV — What Would It Take to Support?
💡
This is Phase E+ territory
CalDAV requires server infrastructure on both sides. It's not a file format — it's a live sync protocol. Complex to build and maintain. Focus on ICS export first (Phase D), then CalDAV later.
What is CalDAV?
WebDAV extension for calendars. Live two-way sync protocol. Every change propagates instantly to the server and to all connected clients.
Client → HTTP PUT /calendars/uid.ics
Server → 200 OK + updated ETAG
Client → HTTP GET /calendars/ → sync
Who uses it?
Apple iCloud Calendar — every Apple device syncs via CalDAV
FastMail Calendar — privacy-focused email/calendar
ownCloud / Nextcloud — self-hosted
Yahoo Calendar (legacy)
macOS Calendar / iOS Calendar — built-in clients
What we'd need to build
1. HTTP client — fetch/put .ics files over HTTPS
2. ETag tracking — detect if server version changed
3. Sync engine — diff local vs server, resolve conflicts
4. Auth handler — OAuth2 or Basic Auth per server
5. Server discovery — /.well-known/caldav endpoint

Apple iCloud — OAuth flow
Apple requires OAuth 2.0 via their Sign in with Apple system. User taps "Connect" → system OAuth dialog → Handy receives a refresh token → stores it securely → uses it to sync.
Google — OAuth 2.0 (Google Calendar API)
Google requires their own OAuth consent screen. You'd need to register a Google Cloud project with "Calendar API" enabled, get a client ID and secret. OAuth flow opens browser → user signs in → Handy gets access token.
Manual CalDAV — URL + username/password
For ownCloud, Nextcloud, FastMail. User enters: https://caldav.fastmail.com/dav/calendars/ + their username/password. Handy tests the connection → stores credentials encrypted.

Ongoing sync
Local change → Server
1. User edits event in Handy
2. Handy PUT /event.ics with new data
3. Server returns 200 + new ETAG
4. Handy stores new ETAG
→ Done: ~200-500ms
Server change → Local
1. Background sync every 60s
2. Handy sends REPORT with local ETAGs
3. Server returns only changed items
4. Handy applies diff
-> Done: real-time-ish
Conflict resolution is the hard part: If you edit an event in Handy AND the server changed it since your last sync, which version wins? Options: server wins, local wins, or show a conflict dialog. Apple Calendar and Google handle this silently (last-write-wins or server-wins). We'd need a strategy.
