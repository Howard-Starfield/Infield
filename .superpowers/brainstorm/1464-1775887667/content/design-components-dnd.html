Sensor split — how card vs column drag is told apart
cardSensor (PointerSensor)
activationConstraint: distance 8px
activator: card body only
→ moves individual card across columns
→ vertical SortableContext handles insertion gap
columnSensor (PointerSensor)
activationConstraint: distance 8px
activator: column header only via dragHandle attr
→ moves entire column (header + all cards)
→ horizontal SortableContext handles column reorder
Both sensors share one DndContext. Active drag type is detected from
active.data.current.type === 'column' | 'card'
DragOverlay renders the appropriate ghost.
⚑ Design flag: color palette vs hex
The backend SelectOption.color currently stores an enum (purple, pink, orange…). A hex string would be more flexible but requires a backend schema change.

Recommendation for Phase A: keep the preset palette (8 colors) — maps directly to existing backend with zero schema change. Phase C can expand to hex if needed.
Spring animation (applied globally)
Cards and columns that shift to make room use:
transition: transform 200ms cubic-bezier(0.34, 1.56, 0.64, 1) On drop, the DragOverlay disappears and the real item snaps in with the same curve — giving the bounce landing feel.
